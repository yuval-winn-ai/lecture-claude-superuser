Skills — Encoded Workflows

Skills are reusable multi-step recipes Claude follows exactly.

# ~/.claude/skills/update-jira/SKILL.md
---
name: update-jira
description: Update Jira ticket fields from GitHub PRs
---
Step 1: Resolve ticket ID from git branch
Step 2: Fetch existing Jira ticket (4 custom fields)
Step 3: Fetch PR data via `gh` CLI
Step 4: Backup original description as Jira comment
Step 5: Synthesize updated Description + Technical Details + QA
Step 6: Update all 3 fields in one API call
Step 7: Confirm with link to ticket

Invoked with: /update-jira <PR_URL> — fire and forget.

Skills save context. This 7-step workflow would require a paragraph of explanation every time. With a skill, it's one command. Perfect for parallel work — type the command in one session, switch to another.

Never Watch Claude Work

The notification pipeline

Hooks Architecture


  Claude finishes task
        │
        ▼
  Stop / Notification hook fires
        │
        ▼
  ~/.claude/hooks/notify.sh
        │
        ├─── Reads JSON payload (cwd, event, message)
        │
        ├─── Detects terminal app (Warp / VS Code / iTerm)
        │
        ├─── Captures active Warp tab title
        │
        ▼
  terminal-notifier
        │
        ├─── macOS notification with sound 🔔
        │
        └─── On click → focus-warp-tab.sh
                  │
                  └─── Cycles Warp tabs (Cmd+Shift+])
                       until title matches → lands on correct tab
Walk through this architecture diagram. The key insight is that clicking the notification takes you to the EXACT tab where Claude was working. This is what makes parallel sessions practical — you don't have to hunt for the right tab.

Hook Configuration

// ~/.claude/settings.json
{
  "hooks": {
    "Notification": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/notify.sh"
      }]
    }],
    "Stop": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/notify.sh"
      }]
    }]
  }
}

Notification — Claude asks a question or needs input
Stop — Claude finished its task

notify.sh — The Core

#!/usr/bin/env bash
PAYLOAD=$(cat)  # Claude passes JSON via stdin

# Extract event type and working directory
CWD=$(echo "$PAYLOAD" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('cwd',''))")
EVENT=$(echo "$PAYLOAD" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('hook_event_name',''))")

case "$TERM_PROGRAM" in
  WarpTerminal)
    # Capture tab title NOW (while in the right tab)
    TAB_TITLE=$(osascript -e 'tell application "System Events" \
      to tell process "Warp" to get name of front window')
    # Save to temp file, pass to focus script on click
    echo "$TAB_TITLE" > "$TMPFILE"
    terminal-notifier -title "Claude Code" \
      -message "$MESSAGE" -sound "Glass" \
      -execute "focus-warp-tab.sh '$TMPFILE'"
    ;;
esac
Walk through the key parts: reading the JSON payload, detecting the terminal, capturing the tab title, and passing it to the focus script via temp file.

focus-warp-tab.sh

#!/usr/bin/env bash
# Called when notification is clicked
TARGET=$(cat "$TMPFILE")

# Bring Warp to front
osascript -e 'tell application "Warp" to activate'

# Cycle tabs until we find the right one
for _ in $(seq 1 30); do
  osascript -e 'tell application "System Events" \
    to tell process "Warp" \
    to keystroke "]" using {command down, shift down}'
  sleep 0.05
  current=$(get_active_tab_title)
  [[ "$current" == "$TARGET" ]] && exit 0
done
Click notification → land on the exact tab
where Claude finished working.

Git Worktrees

The parallel multiplier

The Problem with Parallel Sessions

2 Claude sessions on the same repo = conflict
Both editing the same files, same working tree, git fights

Solution: Git Worktrees

# Separate checkout, shared .git — no clone overhead
git worktree add ../my-repo-fix-login WINN-1234-fix-login
git worktree add ../my-repo-new-api  WINN-1235-new-api

  ~/repos/my-repo/              ← main branch, Claude Session A
  ~/repos/my-repo-fix-login/    ← feature branch, Claude Session B
  ~/repos/my-repo-new-api/      ← feature branch, Claude Session C
           │
           └── All share the same .git (no duplication)
