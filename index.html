Everything Keys Off the Working Directory

Claude's CWD is the root of almost everything it knows about your project.

.claude/settings.json Plan mode, permissions, hooks apply Project defaults silently discarded
.mcp.json Project MCP servers available Claude can't reach the project's DB/API

Walk down the table row by row. Make MCP vivid — the session can't reach MongoDB because .mcp.json didn't load. Memory too — a session that "remembers" in the right CWD writes to an orphan namespace in the parent. The git row is the one that bites fastest: gh pr view fails, and nobody thinks to check CWD. This table is the evidence behind the "1 Claude = 1 project root" rule. Everything else in §1 builds on this.

The VSCode Parent-Folder Trap

The extension uses VSCode's workspace root as Claude's CWD.
Open VSCode on a parent folder → everything on the previous slide breaks.

Common habit

~/work/
├── repo-a/    ← real project
├── repo-b/
└── repo-c/

code ~/work     ← opens parent

CWD = ~/work. No repo-level CLAUDE.md, memory writes to parent namespace, Glob scans all three.

Fixes (in order)

  1. One VSCode window per repo — cleanest
  2. Multi-root workspace — each repo as its own root, so Claude anchors correctly
  3. VSCode as viewer, Warp for Claude — scales best to many parallel sessions
If people remember one thing from §1, this is it. Multi-root workspaces are the decent compromise for VSCode addicts, but once you have 5+ parallel sessions, separate Warp tabs each cd'd into their repo wins every time. This is the moment to lean into the Warp-promotion argument: the terminal is not a legacy tool here, it's the native UI for running many Claude sessions in parallel.

The Silent Failure Mode

Claude won't warn you.
Nothing says "your CLAUDE.md wasn't loaded" or "this MCP server is missing."

Symptoms

  • Claude uses curl when your CLAUDE.md said "always use gh"
  • Claude asks questions already answered by a project skill
  • Claude can't find files that exist in the repo (wrong glob root)
  • /update-jira returns "unknown skill"
  • Past-session memory mysteriously missing
  • You keep repeating yourself — and wonder why

How to verify CWD

  • First turn: "What's your CWD? Which CLAUDE.md files loaded?"
  • Status line — show CWD in your config
  • /status command
  • Check the working-directory line in the system prompt (Ctrl+O transcript)
This is the most dangerous property. No error. No warning. Just quiet degradation and you repeating instructions you thought were baked in. Train the audience: at the start of any real work, ask Claude what its CWD is and what CLAUDE.md files loaded. Make this a reflex.

When a Session Spans Multiple Projects

Real example: change a backend API → update the frontend client → bump the shared types package.

The "1 Claude = 1 project root" rule needs a deliberate exception.
Not "launch from the parent" — anchor + extend.

Wrong

cd ~/work         # parent
claude            # CWD = ~/work
# lose everything from slide 1.4

Right

cd ~/work/backend     # primary
claude \
  --add-dir ../frontend \
  --add-dir ../shared-types
Microservices setups hit this constantly. Temptation: launch from parent. Resist. Next slide shows the right shape — pin CWD to one project, grant explicit access to peers.

--add-dir — Anchor Primary, Extend Scope

CWD stays anchored to the primary; peers are explicitly granted.

One-shot flag

cd ~/work/backend
claude \
  --add-dir ../frontend \
  --add-dir ../shared-types

Persistent (per-project)

// backend/.claude/settings.json
{
  "additionalDirectories": [
    "../frontend",
    "../shared-types"
  ]
}
What stays correct: primary's CLAUDE.md, .mcp.json, skills, memory namespace, git context.
What's bounded: Glob / Grep / Read / Bash see only listed dirs — not the 20-repo parent.
Mental model: you're not moving CWD up, you're adding peer dirs to scope. Primary's config continues to apply. Peers are read/write accessible but aren't treated as projects themselves. You'll see this in my live setup — the main project has claude-dashboard as an additional dir.

Picking the Primary + Orienting Claude

Which repo is "primary"?

  1. Where the change originates
    (defines the API, the types, the contract)
  2. Richest .claude/ config
    (you want its CLAUDE.md + skills loaded)
  3. Where most edits will land

Orient Claude up front

Claude doesn't auto-load CLAUDE.md from added dirs — point at them.

We're working across 3 repos:
  backend (CWD) — defines the API
  frontend — consumes it
  shared-types — the contract

Read ../frontend/CLAUDE.md and
../shared-types/CLAUDE.md before
making any changes.
Two decisions: which repo is primary, and how you brief Claude about the others. Orientation matters — Claude will happily edit frontend without ever reading its CLAUDE.md unless you tell it to. If you do cross-repo work often, save this orientation as a skill (e.g. /cross-repo-orient) instead of retyping.

Cross-Repo Guardrails

Multi-repo sessions are the highest-blast-radius work. Tighten the controls.

Plan mode Non-negotiable. Plan first, review, then approve.
Budget context Peers expand search scope. Point Claude at specific files — line ranges over @file.
One branch name Same Jira ticket / branch name in every repo.
Explicit git paths git -C ../frontend … for peers — or let Claude resolve paths.
Watch more, not less Less autonomy than single-repo. Fewer sessions, closer watch.
One PR per repo Draft linked PRs in each repo — reviewable atomically.
Theme: cross-repo sessions earn less autonomy, not more. Plan mode gate, explicit context hints, closer supervision. Also the strongest case against parallelism — don't try to run 3 cross-repo sessions at once.

When Not to Go Multi-Repo

Skip multi-repo when…

  • Changes can be sequenced — two clean sessions beat one tangled one.
  • You're only reading or exploring — single-repo session or Chat. Multi-repo earns its cost only when you're writing across boundaries.
  • "Shared" is a published dep — update, publish, consume. Don't patch in-place just because Claude can.

Umbrella folder
(frequent cross-cutting work)

~/work/cross-cutting/
├── CLAUDE.md            ← architecture notes
├── .claude/
│   ├── settings.json    ← additionalDirectories
│   └── skills/          ← cross-repo skills
└── (no code — launch point)

An explicit, named place for cross-repo work —
so you stop drifting back to the parent folder.

Sequence-it is often cleaner than people think. Two focused sessions beat one tangled multi-repo session almost every time. But when the change must land together, the umbrella folder is the permanent home — give it its own CLAUDE.md describing system boundaries, and add skills tuned to cross-cutting work. It turns "drift back to the parent" into "deliberate cross-cutting mode."

Plan Modelive demolaptop

With 6 sessions running, the bottleneck is your attention, not Claude's speed.

Plan Mode — default

  • Review a 1-page plan, not a 400-line diff
  • Catch wrong approach before the cascade
  • Sessions block waiting for you — no races
  • Safe to leave unattended

Accept Edits — opt in

  • Narrow, well-scoped tasks
  • Reversible work in a throwaway worktree
  • Sessions you're actively watching
  • Shift+Tab to switch per-session

What happens when there's too much context
Common misconception: the VSCode extension ships the whole file on every turn. It doesn't — it sends a lightweight pointer. Claude decides what to Read. The @ shortcut is the wasteful pattern — it inlines the full file. Use line ranges or let the IDE pointer do the work.

Why Token Waste Hurts

One session's bloat taxes the whole fleet.

CostWhat actually happens
Usage limitsFewer parallel sessions before you're rate-limited
LatencyLonger turns × many sessions = fewer iterations per hour
Attention dilutionQuality drops before the window fills — key details get "lost in the middle"
Forced compactionClaude summarizes older turns — exact paths, line numbers, and reasoning get lost
Cache bustingNoisy tool output breaks the prompt cache — every turn after costs more
Stall riskBloated sessions stall mid-task — rescue costs more than scoping to finish clean
Short, focused sessions aren't just safer — they're strictly higher throughput.
The obvious costs — usage limits, latency — are the ones everyone names first. The less obvious ones matter more: - Attention dilution: model quality degrades with noise well before hitting the window. - Compaction is lossy. Post-compaction sessions often repeat mistakes the pre-compaction session already learned from. - Cache busting: Claude Code caches the conversation prefix. Churn in that prefix makes every turn pay the miss. - Fleet tax: every extra second per turn compounds across 5+ parallel sessions. This slide is the *why* behind the next slide's "many short sessions" recommendation — and the *prerequisite* for §7 parallel work.

Coming Up

This is the section where the audience starts saving real time. Skills especially — /update-jira is the canonical example. Frame it as: you're programming Claude's defaults, not its behavior.

Update Claude's Settings

If Claude doesn't follow your instructions
more than once — don't just repeat yourself.

Update its configuration to make it permanent.

  • ~/.claude/settings.json — hooks, permissions, plugins, default mode
  • ~/.claude/CLAUDE.md — persistent behavioral instructions
  • .claude/settings.json — per-project overrides
  • /update-config — ask Claude to update its own config
Key insight: Claude reads these files every session. If you keep telling Claude the same thing, make it a permanent instruction. Especially important for parallel work — you don't want to repeat context in every session.

CLAUDE.md — Persistent Instructions

# ~/.claude/CLAUDE.md (global — every session reads this)

## GitHub Access
When accessing GitHub content (PRs, issues, code, etc.),
always use the `gh` CLI instead of WebFetch or browser tools.

## Jira Field Reference (winnai.atlassian.net)
The QA Instructions field is `customfield_10051`.
Always request it explicitly by key when fetching Jira issues.

Global: ~/.claude/CLAUDE.md — applies to all sessions
Project: .claude/CLAUDE.md — repo-specific rules

This is the real CLAUDE.md from my setup. Every session reads this automatically. No need to re-explain GitHub or Jira conventions — it's baked in.

The notification pipeline

Coming Up

This is the section with the signature demo. The hook pipeline is the mechanical answer to "how do I run 5 sessions without going crazy?" Build anticipation — the demo at the end is the payoff.

Hooks Architecture


  Claude finishes task
        │
        ▼
  Stop / Notification hook fires
        │
        ▼
  ~/.claude/hooks/notify.sh
        │
        ├─── Reads JSON payload (cwd, event, message)
        │
        ├─── Detects terminal app (Warp / VS Code / iTerm)
        │
        ├─── Captures active Warp tab title
        │
        ▼
  terminal-notifier
        │
        ├─── macOS notification with sound 🔔
        │
        └─── On click → focus-warp-tab.sh
                  │
                  └─── Cycles Warp tabs (Cmd+Shift+])
                       until title matches → lands on correct tab
Walk through the architecture. Key insight: clicking the notification takes you to the EXACT tab where Claude was working. This is what makes parallel sessions practical — you don't have to hunt for the right tab.

Hook Configuration

// ~/.claude/settings.json
{
  "hooks": {
    "Notification": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/notify.sh"
      }]
    }],
    "Stop": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "~/.claude/hooks/notify.sh"
      }]
    }]
  }
}

Notification — Claude asks a question or needs input
Stop — Claude finished its task

notify.sh — The Core

#!/usr/bin/env bash
PAYLOAD=$(cat)  # Claude passes JSON via stdin

# Extract event type and working directory
CWD=$(echo "$PAYLOAD" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('cwd',''))")
EVENT=$(echo "$PAYLOAD" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('hook_event_name',''))")

case "$TERM_PROGRAM" in
  WarpTerminal)
    # Capture tab title NOW (while in the right tab)
    TAB_TITLE=$(osascript -e 'tell application "System Events" \
      to tell process "Warp" to get name of front window')
    # Save to temp file, pass to focus script on click
    echo "$TAB_TITLE" > "$TMPFILE"
    terminal-notifier -title "Claude Code" \
      -message "$MESSAGE" -sound "Glass" \
      -execute "focus-warp-tab.sh '$TMPFILE'"
    ;;
esac
Walk through: reading the JSON payload, detecting the terminal, capturing the tab title, passing it to the focus script via temp file.

focus-warp-tab.sh

#!/usr/bin/env bash
# Called when notification is clicked
TARGET=$(cat "$TMPFILE")

# Bring Warp to front
osascript -e 'tell application "Warp" to activate'

# Cycle tabs until we find the right one
for _ in $(seq 1 30); do
  osascript -e 'tell application "System Events" \
    to tell process "Warp" \
    to keystroke "]" using {command down, shift down}'
  sleep 0.05
  current=$(get_active_tab_title)
  [[ "$current" == "$TARGET" ]] && exit 0
done
Click notification → land on the exact tab
where Claude finished working.
